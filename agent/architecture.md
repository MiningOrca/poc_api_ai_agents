# Architecture

## Purpose

This project is a layered QA automation pipeline that transforms business context and an API contract into test artifacts, execution plans, execution results, and structured review output.

The system is intentionally split between **skill-driven reasoning layers** and **deterministic runtime layers**.

The core idea is simple:

- use LLM skills where interpretation, planning, and judgment are genuinely useful;
- use regular code where reproducibility, validation, and runtime behavior must be strict.

This is not a generic chat assistant. It is an artifact-oriented pipeline for API QA automation.

---

## Inputs

Primary source inputs:

- `agent/input/context.md` — natural-language business and product rules
- `agent/input/open_api.json` — OpenAPI / Swagger contract

Reference artifact examples for output shaping:

- `agent/json_examples/rules.json`
- `agent/json_examples/test_cases.json`
- `agent/json_examples/execution_plan.json`
- `agent/json_examples/execution_report.json`
- `agent/json_examples/review_report.json`

These example files are the current contract seeds for prompt shaping and output normalization not a formal json schemas. Generate schemas if needed.

---

## Scope

In scope:

1. extract rules from context
2. design test cases per endpoint
3. generate execution plans with setup chains and context passing
4. execute plans through a standard executor
5. review execution output to classify failures and assess validation quality

Out of scope for this architecture version:

- CI/CD wiring
- environment provisioning
- flaky test recovery strategies
- long-term test scheduling
- generic autonomous debugging loops

---

## Architectural principles

### 1. Layered responsibility

Each stage must have a single clear responsibility:

- rules extraction extracts intent
- test design decides what should be tested
- execution planning decides how the test should be executed
- executor performs runtime actions
- review analyzes what happened and whether the test was good enough

### 2. Skills for reasoning, code for enforcement

LLM skills are used where the pipeline needs semantic interpretation or planning. Regular code is used where the pipeline needs:

- strict validation
- repeatable execution
- transport-level behavior
- stable serialization
- runtime error handling

### 3. Deterministic gates between reasoning layers

Every skill output is treated as a **candidate artifact**, not as automatically trusted truth.

Before any candidate artifact can move to the next stage, deterministic gates must validate it. This gate layer exists even if the implementation is generated by Claude later.

### 4. Traceability

Artifacts must preserve traceability from source rules to test cases and from test cases to execution and review. The system should always be able to answer:

- which business rule a test covers
- which step failed
- whether a failure is likely caused by the application, the test, the contract, or the environment

### 5. Minimal but explicit contracts

Artifacts should stay compact, but every boundary should still have an explicit shape. The first implementation can use example-driven contracts plus strict typed validation. The architecture does not rely on free-form text between stages.

## High-level pipeline

The pipeline consists of five runtime levels and four skill-driven roles.

### Level 1 — Rules Extraction

**Input:** `agent/input/context.md`  
**Output:** `rules.json`  
**Driver:** Skill

This layer reads the business context and extracts:

- global rules
- endpoint-specific rules
- references to source sections

This artifact becomes the business-facing truth for downstream generation.

### Level 2 — Test Case Design

**Input:** `rules.json` + `agent/input/open_api.json`  
**Output:** `test_cases.json`  
**Driver:** Skill

This layer designs test intent per endpoint. It decides:

- what should be tested
- whether the case is positive / negative / boundary
- whether the flow is single-step or chained
- which endpoint-level steps are needed at a design level
- which source rules justify the case

This layer does **not** decide concrete HTTP payloads or transport-level assertions.

### Level 3 — Execution Planning

**Input:** `test_cases.json` + `agent/input/open_api.json` + reference examples  
**Output:** `execution_plan.json`  
**Driver:** Skill

This layer converts abstract cases into executable scenarios.

This stage is intentionally skill-driven rather than purely deterministic because API testing often needs:

- multi-step setup chains
- state transitions across endpoints
- context propagation from prior responses
- planning around business preconditions

A rigid compiler may become brittle for these cases. Instead, the skill produces a concrete execution candidate, and deterministic validation then checks it.

### Level 4 — Execution

**Input:** `execution_plan.json`  
**Output:** `execution_report.json`  
**Driver:** Code

This layer performs runtime execution. It:

- sends HTTP requests
- resolves bindings from shared context
- records actual request / response data
- evaluates assertions
- stores step-level and scenario-level results

The executor is normal application code, not a skill.

### Level 5 — Result Review

**Input:** `execution_report.json` + `execution_plan.json` + `rules.json` + relevant contract slice  
**Output:** `review_report.json`  
**Driver:** Skill

This layer reviews failures and insufficient validation. It does not re-execute anything. It answers:

- why the scenario failed
- whether the failure is likely an application issue, test issue, contract mismatch, or environment issue
- whether the executed assertions were sufficient
- what additional validation would have improved confidence

## Skill set

The system uses four main skills.

### 1. Rules Extraction Skill

**Responsibility**

Transform `context.md` into a compact and traceable rules artifact.

**Output expectations**

- separate general rules from endpoint-specific rules
- keep rules concise and atomic
- keep source references when possible
- do not invent endpoints or validations not present in the source

### 2. Test Design Skill

**Responsibility**

Transform rules plus contract awareness into endpoint test cases.

**Output expectations**

- generate concise, business-grounded cases
- categorize cases as `positive`, `negative`, or `boundary`
- use source references to explain why each case exists
- list only high-level steps, not transport-level request details

### 3. Execution Planning Skill

**Responsibility**

Transform test cases into executable scenarios.

**Output expectations**

- decide setup steps and target steps
- resolve endpoint sequence
- generate concrete method, path, params, body, assertions, and bindings
- produce plans that match the API contract
- use only explicitly supported assertion operators

### 4. Result Review Skill

**Responsibility**

Review execution outcomes and evaluate diagnostic quality.

**Output expectations**

- classify failure cause
- explain evidence behind the verdict
- assess whether validation was sufficient
- recommend concrete next actions

## Deterministic gates

Even though three major generation layers are skill-driven, the system still requires mandatory deterministic gates.

These gates are not optional. They are what prevent the architecture from becoming prompt-only logic.

### Gate A — Artifact shape validation

Every candidate artifact must be validated against a strict internal model.

The first implementation may use typed code models seeded from the example files in `agent/json_examples`. Formal JSON Schemas may be introduced later, but the gate behavior must already exist now.

This gate should reject:

- missing required fields
- unknown top-level sections when not allowed
- wrong primitive types
- malformed nested objects

### Gate B — Contract conformance validation

Artifacts that reference runtime API behavior must be validated against `open_api.json`.

This gate should verify:

- endpoint IDs map to real operations
- HTTP methods are valid
- paths exist
- path/query/body fields are supported
- expected status codes are documented
- response assertions only target plausible fields

### Gate C — Assertion operator validation

The initial supported assertion operators are:

- `equals`
- `not_equals`
- `exists`
- `not_exists`
- `contains`
- `in`
- `gte`
- `lte`

No additional operators should be accepted in the first version.

### Gate D — Binding validation

Bindings must be structurally and semantically valid.

The validator should check:

- `produceBinding` / `produceBindings` use legal source paths
- consumed context keys are defined earlier in the scenario or in setup context
- no step consumes values that were never produced
- key names are stable and predictable across the scenario

### Gate E — Normalization

Accepted artifacts should be normalized into a canonical internal representation before they are persisted or executed.

Normalization may include:

- stable field ordering
- defaulting empty collections
- filling omitted optional structures with canonical empty values
- canonical boolean / numeric formatting where needed

## Artifact boundaries

The pipeline has five main artifact boundaries.

### 1. `rules.json`

Purpose:

- store general rules
- store endpoint-specific rules
- keep references to source material

Minimum conceptual structure:

- `generalRules`
- `rulesByEndpoint`

Recommended enrichment:

- rule ID
- source references
- concise atomic rule text

### 2. `test_cases.json`

Purpose:

- store design-level test intent per endpoint

Typical content:

- `endpointId`
- case list
- `title`
- `summary`
- `category`
- `sourceRefs`
- design-level `steps`

Recommended enrichment:

- `mode` such as `single` or `chain`
- `expectedStatusCode` when sufficiently grounded
- compact expected outcome text

### 3. `execution_plan.json`

Purpose:

- store executable scenarios for the executor

This is the most operational artifact. It should contain scenario-level metadata and step-level transport details.

Each step may include:

- `endpointId`
- `title`
- `stepRole`
- `method`
- `path`
- `pathParams`
- `queryParams`
- `body`
- `assertions`
- `produceBindings`
- optional consume binding representation
- `expectedStatusCode`

### 4. `execution_report.json`

Purpose:

- store the normalized runtime result

This report must be sufficient for downstream review without needing raw executor logs.

It should include:

- scenario metadata
- overall pass/fail
- start/end timestamps
- failed step index when applicable
- per-step request/response summary
- expected vs actual status code
- assertion result list
- execution errors

### 5. `review_report.json`

Purpose:

- store a structured post-execution diagnosis

It should include:

- verdict
- summary
- failure classification
- validation sufficiency assessment
- root-cause hypotheses
- evidence
- recommended actions

## Execution planning model

Execution planning is the most important reasoning layer in this architecture.

The planner must be able to represent both simple and chained scenarios.

### Single-step scenario

A single-step scenario has one target operation and no setup chain.

### Chained scenario

A chained scenario may include:

- one or more setup steps
- intermediate context extraction
- one final target step
- assertions across both setup and target operations where useful

The planner is expected to decide when state must be created first rather than assuming all cases are standalone.

### Bindings

Bindings allow state to flow from one step to another.

A typical example is extracting a user identifier from one response and using it in a later request path or body.

The architecture treats bindings as first-class execution-plan data rather than hidden executor behavior.

### Assertions

Assertions are intentionally limited to a small initial operator set. This keeps executor behavior simple and review output interpretable.

The first version should prefer simple, composable assertions rather than a large DSL.

---

## Review model

The review layer is not a free-form narrative. It is a structured diagnostic stage.

The reviewer should be able to classify at least these kinds of outcomes:

- application bug
- test issue
- contract mismatch
- environment issue
- inconclusive

The reviewer must also assess validation sufficiency. A scenario can fail for one reason and still have weak validation. Those are separate dimensions.

Example review questions:

- Did the target step fail because the application returned an unexpected status code?
- Did the test assert too little to prove the intended business rule?
- Did the scenario rely on unstable or missing bindings?
- Did the observed result contradict the documented contract?

## Why this architecture

This architecture intentionally avoids two bad extremes:

### Extreme 1 — everything deterministic

This usually breaks down once execution planning needs real state transitions, multi-endpoint setup, or context propagation.

### Extreme 2 — everything prompt-driven

This creates a system that is hard to trust, hard to validate, and hard to debug.

The chosen design is a middle path:

- interpretation and planning are skill-driven
- validation and execution are deterministic
- review is skill-driven again, but only after runtime facts are captured in a strict report

That split is the main architectural decision of this project.

---

## Final summary

The system is a five-level QA automation pipeline with four skill-driven reasoning roles.

- `context.md` becomes `rules.json`
- rules plus API contract become `test_cases.json`
- test cases become `execution_plan.json`
- executor produces `execution_report.json`
- reviewer produces `review_report.json`

The key design decision is that execution planning is skill-driven because test execution often depends on non-trivial state transitions and context passing. However, every skill output must pass deterministic validation and normalization before the next stage can trust it.

This architecture preserves flexibility where reasoning is required and strictness where runtime behavior must be dependable.
